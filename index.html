推理基础

从 Transformer 到自回归生成
Part 1 开始。主讲人 A 接力。

从序列模型到 Attention

大家好，欢迎来到今天的课堂。今天我们的主题是大语言模型的推理，内容分为三个部分：首先我会带大家回顾 Transformer 的推理流程，然后第二部分介绍推理加速技术，第三部分探讨一个有趣的问题——能不能通过增加推理时的计算量来提升模型效果。

先来看这张对比图。左边是我们熟悉的 RNN，处理序列信息时必须一个一个地传递，信息从 $h_1 \to h_2 \to h_3$，依次串行传递。这就像传话游戏，第一个人的信息要一个个传到最后，中间很容易丢失，训练时还会遇到梯度消失的问题，长距离依赖关系很难学到。

右边是 Transformer 的核心创新——Attention 机制。它的关键在于：每个位置可以直接"看到"所有其他位置，不需要串行传递。就像所有人同时在一个会议室里，谁想说什么直接说，不需要传话。这种并行性是后面推理加速的重要基础。

Self-Attention 机制

核心公式：

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$
  • Q (Query): 当前 token 想了解什么
  • K (Key): 每个 token 能提供什么信息
  • V (Value): 每个 token 的实际内容
  • $\sqrt{d_k}$ 缩放防止 softmax 饱和
接下来我们详细拆解 Attention 的计算过程。请看屏幕上的公式 $\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$。

为了方便理解，我们用一个图书馆的类比。左边这张流程图下面也画了这个类比。想象你去图书馆找资料，你在搜索框输入的关键词就是 Query——你想了解什么。每本书上贴的标签和索引就是 Key——这本书能提供什么信息。书的实际内容就是 Value。

计算过程是这样的：首先 $Q$ 和 $K$ 做内积，得到一个"相关性分数"——你的关键词和每本书的标签有多匹配。然后过一遍 softmax，把这些分数变成权重，加起来等于 1。最后用这些权重对 $V$ 做加权求和，得到最终输出。

公式里除以 $\sqrt{d_k}$ 这一步很重要：当维度 $d_k$ 比较大时，$QK^T$ 的内积数值会很大，导致 softmax 输出接近 one-hot，梯度几乎为零。除以 $\sqrt{d_k}$ 可以缓解这个问题。

Multi-Head Attention

为什么不只用一个 Attention？

  • 每个 Head 学习不同的注意力模式
  • Head 1 可能关注语法关系
  • Head 2 可能关注语义相似性
  • Head 3 可能关注位置关系...
关键参数

LLaMA-2-7B: 32 heads, $d_{model}=4096$, $d_{head}=128$

理解了单个 Attention 之后，一个自然的问题是：为什么不只用一个？为什么要搞多头？

答案在左边这张图上。每个 Head 有自己独立的 $Q$、$K$、$V$ 权重矩阵，独立计算 Attention，最后把所有头的结果拼接起来，通过一个线性层投影回去。

关键在于：不同的 Head 会学到不同的注意力模式。比如 Head 1 可能关注语法关系，Head 2 关注语义相似性，Head 3 关注位置关系。通过多个"视角"来看同一句话，模型能捕获更丰富的信息。

举一个具体的例子：LLaMA-2-7B 有 32 个头，每个头的维度是 128，$32 \times 128 = 4096$ 正好等于模型的隐藏维度 4096。这个设计在后续理解推理的计算量时会用到。

Transformer Decoder Block

现在我们把 Attention 放到完整的 Transformer Decoder 里。请看这张结构图，从下往上看。

最下面是输入经过 Embedding 和位置编码。然后进入 Masked Multi-Head Self-Attention，注意这里是 Masked 的——每个位置只能看到自己和之前的位置，不能"偷看"后面的 token。这是因果性约束，也是后面自回归生成的基础。

接下来是 Add and LayerNorm——残差连接加层归一化。残差连接把输入直接加到输出上，缓解了深层网络的梯度消失问题。然后是 Feed-Forward Network，也就是 FFN，由两个线性层和一个激活函数组成，存储了模型大部分的参数。

这个 Attention 加 FFN 的结构会在上方虚线框里重复堆叠。比如 LLaMA-2-7B 就堆叠了 32 层，每一层都有自己独立的参数。最终通过一个线性层和 Softmax，输出词表上的概率分布。

自回归生成 (Autoregressive Generation)

核心思想：逐 token 生成，每次根据已有内容预测下一个

$$P(y_t \mid y_{\lt t}, x) = \text{softmax}(W \cdot h_t)$$
  • $x$ = 输入 prompt
  • $y_{\lt t}$ = 已生成的 tokens
  • $h_t$ = Decoder 最后一层的隐状态
  • 每次只生成一个 token，追加后继续
  • 遇到 [EOS] 或达到长度上限时停止
为什么慢？

生成 1000 个 token 需要跑 1000 次前向传播！

有了 Transformer Decoder 的结构，我们来看它是怎么生成文字的。

核心思想很简单：逐个 token 生成，每次根据已有的内容预测下一个。请看左边的示意图：第一步输入起始符 BOS，模型预测出"今"字。第二步把"今"追加进去，模型预测"天"。第三步继续，直到遇到结束符或者达到长度上限。

右边是这个过程的数学表达。$h_t$ 是 Decoder 最后一层输出的隐状态，通过一个线性层加 softmax，得到词表上的概率分布。选择概率最大的或者采样的 token 作为下一个输出。

请大家注意右下角这个提示：生成 1000 个 token 需要跑 1000 次前向传播。这就是 LLM 推理慢的根本原因——每生成一个词，整个模型都要完整跑一遍。记住这一点，后面讲加速技术时会用到。

Prefill vs Decode：推理的两个阶段

进一步细化推理过程，我们可以把它分成两个阶段：Prefill 和 Decode。

请看左边，Prefill 阶段。用户的整个 prompt 一次性送入模型，做的是矩阵乘法——一个 $N \times d$ 的输入矩阵乘以 $d \times d$ 的权重矩阵。这是典型的计算密集型操作，GPU 的算力可以充分利用。

再看右边，Decode 阶段。每次只有一个新 token 进来，做的是向量乘法——一个 $1 \times d$ 的向量乘以 $d \times d$ 的权重矩阵。计算量很小，但每次都要从显存里读取整个权重矩阵。所以瓶颈不在算力，而在显存带宽——这就是 Memory-bound。

打个比方：Prefill 像是快递员一次性送一批快递，效率很高。Decode 像是每次只送一个快递但都要跑一趟仓库取货，大部分时间花在路上。

KV Cache：避免重复计算

接下来看推理加速最基本也最重要的技术——KV Cache。

请看左半边，没有 KV Cache 的情况。$t=1$ 时计算 token a 的 $K$ 和 $V$。$t=2$ 时要把 a 和新的 token b 一起重新算 $K$ 和 $V$。$t=3$ 又要把 a、b、c 全部重算。虚线框里标红的都是重复计算——之前算过的 $K$ 和 $V$，每次又要重新算一遍，非常浪费。

右边是有了 KV Cache 的做法。$t=1$ 算出 $K_1$ 和 $V_1$，存入 cache。$t=2$ 只需要算新 token 的 $K_2$ 和 $V_2$，然后从 cache 里读取之前的 $K_1$ 和 $V_1$。$t=3$ 同理，只算 $K_3$ 和 $V_3$，从 cache 读取之前所有的 $K$ 和 $V$。

Trade-off 很明显：KV Cache 节省了大量重复计算，加速了推理，但代价是占用额外的显存来存储这些 $K$ 和 $V$ 向量。具体占多少呢？下一张 slide 来量化分析。

KV Cache 占多少显存？

$$M_{\text{KV}} = 2 \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times S \times \text{bytes}$$
参数含义LLaMA-2-7B
$n_{\text{layers}}$层数32
$n_{\text{heads}} \times d_{\text{head}}$注意力维度4096
$S$序列长度变量
bytes精度 (FP16)2
计算示例

LLaMA-2-7B, $S=4096$:
$M = 2 \times 32 \times 4096 \times 4096 \times 2$ $= 2\,\mathrm{GB}$

（图表由 Python 生成。模型参数来自 Touvron et al. 2023 / HuggingFace LlamaConfig）

KV Cache 到底占多少显存？请看这个公式。$M_{\text{KV}}$ 等于 $2 \times n_{\text{layers}} \times d_{\text{head}} \times n_{\text{heads}} \times S \times \text{bytes}$。其中 $2$ 是因为要同时存 $K$ 和 $V$ 两份。

我们用 LLaMA-2-7B 的具体参数来算。$n_{\text{layers}}=32$，注意力维度 $4096$，FP16 精度下每个元素 $2$ 字节。当序列长度 $S=4096$ 时，代入公式：$2 \times 32 \times 4096 \times 4096 \times 2 = 2\,\text{GB}$。

也就是说，仅仅是为了存储 KV Cache，序列长度 $S=4096$ 时就要 $2\,\text{GB}$ 显存。如果换成 70B 的模型，或者更长的序列，这个数字会更加夸张。

右边的柱状图直观展示了不同序列长度下的显存占用。大家可以看到，长文本场景下 KV Cache 会成为显存瓶颈。这就是为什么后面会有 PagedAttention 等技术来优化 KV Cache 的管理。

推理延迟分解

$$T_{\text{total}} = T_{\text{prefill}} + N \times T_{\text{decode}}$$
  • TTFT (Time To First Token) $= T_{\text{prefill}}$
    用户发 prompt 到看到第一个 token 的时间
  • TPOT (Time Per Output Token) $= T_{\text{decode}}$
    每生成一个 token 的平均时间
瓶颈在哪里？

对于长生成（$N$ 大），Decode 阶段 dominates，因为 Decode 是 memory-bound（受限于显存带宽）

（图表由 Python 生成，延迟值基于 LLaMA-2-7B FP16 在 A100 上的典型推理延迟）

有了 Prefill/Decode 和 KV Cache 的概念，我们可以精确地分析推理延迟了。

总延迟 $T_{\text{total}} = T_{\text{prefill}} + N \times T_{\text{decode}}$，这里 $N$ 是输出 token 的数量。两个关键指标：TTFT 也就是 Time To First Token，$= T_{\text{prefill}}$，代表用户发出 prompt 后等多久才能看到第一个字。TPOT 也就是 Time Per Output Token，$= T_{\text{decode}}$，代表每生成一个 token 要多久。

关键洞察在右边的瀑布图里。当生成 500 或 1000 个 token 时，Decode 阶段占据了绝大部分时间。因为 Decode 是 Memory-bound 的——每步都要从显存读取整个权重矩阵，但只做一次很小的向量乘法，GPU 算力大量闲置。

所以加速推理的重点，在于优化 Decode 阶段的 Memory-bound 问题。具体怎么优化呢？这就是 Part 2 要讲的内容。

LLM 推理全流程：从输入到输出

这张 slide 把前面讲的所有概念串成了一个完整的推理流程图，大家跟着我从左到右看一遍。

用户输入一段自然语言的 prompt，首先经过 Tokenization 转换成 token ID 序列。然后进入 Prefill 阶段，并行处理所有 prompt token，这一步是 Compute-bound 的矩阵乘法。接着进入 Decode Loop，逐个 token 生成，每步都是 Memory-bound 的向量乘法，KV Cache 逐步增长。生成结束后，经过 Detokenization 把 token ID 转回文本输出。

下方标注了三个关键性能指标：TTFT、TPOT 和总延迟。对于长生成场景，Decode 阶段 dominates，加速推理的重点就在这里。底部也预告了下一部分的优化技术：Speculative Decoding、量化、Flash Attention、PagedAttention。

推理加速技术

从瓶颈分析到系统优化
Part 1 结束，过渡到 Part 2。主讲人 B 接力。 过渡语：了解了推理流程和瓶颈，接下来看如何加速这个过程。

推理瓶颈：Memory-bound vs Compute-bound

$$\text{Arithmetic Intensity} = \frac{\text{FLOPs}}{\text{Bytes accessed}}$$
  • Memory-bound: 算术强度低，GPU 等待数据
    → Decode 阶段的 Attention 计算
  • Compute-bound: 算术强度高，GPU 算力饱和
    → Prefill 阶段的大矩阵乘法
核心洞察

LLM 推理的瓶颈在 Decode 阶段的内存带宽，不是算力。加速的关键：减少访存量或提高访存效率。

Roofline Model（图表由 Python 生成。硬件参数参考 NVIDIA A100 80GB SXM4 规格：HBM 带宽 2039 GB/s，FP16 Tensor Core 312 TFLOPS）

大家好，承接刚才的内容，我们知道 LLM 推理的瓶颈在 Decode 阶段的内存带宽。这一页我们从更底层的视角来分析。

Roofline Model 是分析硬件性能瓶颈的经典模型。核心概念是算术强度，等于计算量除以访存量。算术强度低意味着 GPU 大量时间在等数据，这就是 Memory-bound。算术强度高意味着 GPU 算力跑满了，这就是 Compute-bound。

看右边的 Roofline 图。在算术强度较低的区域，性能被内存带宽限制，曲线是上升的。到了算术强度足够高时，性能被算力限制，曲线变成平台。LLM 的 Decode 阶段恰恰落在左边这个上升区域——每个 token 要读取几十 GB 的模型权重，但只做几 GFLOP 的计算，算术强度很低。

所以加速推理有三个核心方向：减少模型大小，也就是量化；减少访存次数，比如 Flash Attention；或者用小模型来帮忙，也就是 Speculative Decoding。接下来逐一介绍。

Speculative Decoding（投机解码）

首先介绍一种非常巧妙的加速方法——Speculative Decoding，投机解码。

核心想法用一个类比来理解：实习生写草稿，主管来审核。实习生相当于一个小模型，比如 $7\text{B}$，虽然不是那么准但生成速度很快。主管相当于大模型，比如 $70\text{B}$，虽然很准但每次推理很慢。

具体流程分三步，请看图。第一步 Draft，小模型快速生成 $K$ 个 token 作为草稿。第二步 Verify，大模型用一次前向传播同时验证这 $K$ 个 token。第三步输出，从左到右逐个检查，小模型猜对的就接受，猜错的就拒绝并重新采样。

图上的例子：$K=3$，小模型生成了 $t_1$、$t_2$、$t_3$ 三个 token。大模型验证后 $t_1$ 和 $t_2$ 通过了，$t_3$ 被拒绝。最终一次循环就产出了 $2$ 个有效 token，相当于 $2\times$ 加速。

最关键的性质是：最终输出的概率分布和大模型完全一致，不会有任何质量损失。这个等价性有严格的数学证明，下一页我们来看。

Speculative Decoding：为什么结果等价？

目标：证明用 Speculative Decoding 采样的 token 分布与直接从大模型 $p$ 采样完全一致。

接受概率
$$p_{\text{accept}}(x) = \min\!\left(1,\; \frac{p(x)}{q(x)}\right)$$

$p$ = 大模型概率，$q$ = 小模型概率
如果 $p(x) \geq q(x)$：必接受
如果 $p(x) < q(x)$：以 $p(x)/q(x)$ 的概率接受

拒绝后重采样
$$p_{\text{resample}}(x) = \frac{\max(0,\; p(x) - q(x))}{\sum_{x'} \max(0,\; p(x') - q(x'))}$$

只从 $p > q$ 的区域重新采样
保证最终分布为 $p$

最终采样分布：

$$P_{\text{final}}(x) = \underbrace{q(x) \cdot \min\!\left(1, \frac{p(x)}{q(x)}\right)}_{\text{被接受}} + \underbrace{\left(1 - \sum_{x'} q(x') \min\!\left(1, \frac{p(x')}{q(x')}\right)\right)}_{\text{拒绝概率}} \cdot \underbrace{p_{\text{resample}}(x)}_{\text{重采样}} = p(x) \;\checkmark$$
期望加速比

若小模型与大模型平均接受率 $\alpha$，draft 长度 $K$，则期望每步输出 token 数：$\frac{1}{1-\alpha}$ 倍

这一页我们证明 Speculative Decoding 的输出分布与大模型完全一致。

左边是接受概率的公式：$p_{\text{accept}}(x) = \min\!\left(1,\; \frac{p(x)}{q(x)}\right)$，其中 $p$ 是大模型的概率，$q$ 是小模型的概率。如果大模型认为这个 token 的概率大于等于小模型的概率，直接接受。如果小模型给出的概率过高，就以 $p(x)/q(x)$ 的概率接受。

右边是被拒绝后的重采样公式：只从大模型概率高于小模型的那些 token 中重新采样，权重是 $p(x) - q(x)$。

下面是完整的推导：最终采样分布等于被接受的部分加上被拒绝后重采样的部分。经过化简，最终结果恰好等于 $p(x)$，也就是大模型本身的分布。这就是为什么 Speculative Decoding 可以保证输出质量不变。

实践中，如果小模型和大模型比较接近，接受率 $\alpha$ 通常在 $0.8$ 到 $0.9$ 之间，这意味着 $3$ 到 $5$ 倍的加速。

上一页证明了数学等价性，这页用图表展示实际加速效果。

横轴是草稿长度 $K$，纵轴是期望加速比，不同颜色的曲线代表不同的接受率 $\alpha$。可以看到，当接受率 $\alpha=0.9$、$K=5$ 时，能获得约 3 倍加速。但 $K$ 继续增大后收益递减，因为 $K$ 个 token 全部正确的概率是 $\alpha^K$，会快速衰减。

实践中的建议：$K$ 通常设为 $4$ 到 $7$，草稿模型选择与大模型同系列的小模型，这样可以获得 $0.9$ 以上的接受率。关键约束是不改变输出分布——这是由数学等价性保证的。

总结一下 Speculative Decoding：无损加速，原理简洁，效果显著。接下来我们看另一个优化方向——量化。

模型量化 (Quantization)

用更少位数表示模型权重，牺牲少量精度换取：

  • 减少显存：$\text{FP16}\to\text{INT4}$，模型体积缩小 $4\times$
  • 加速推理：更小的数据 $\to$ 更少的访存
  • 更大 batch：省下的显存可服务更多请求
精度每参数字节7B 模型大小质量损失
FP162 B14 GB—
INT81 B7 GB极小
INT40.5 B3.5 GB小
量化的核心思想很简单：用更少的位数来表示模型的权重。

请看左边的数轴对比。上面一行是 FP16，也就是半精度浮点数，可以表示 65536 个不同的值，点非常密集。下面一行是 INT4，只有 16 个不同的值，点非常稀疏。量化的过程就是把密集的浮点值映射到稀疏的离散值上，用公式表示就是 $x_q = \text{round}(x/s) + z$，其中 $s$ 是缩放因子，$z$ 是零点。

这就像把一张高清照片压缩成缩略图——肉眼看差别不大，但文件小了很多。

看右边的表格：FP16 时 LLaMA-2-7B 占 $14\,\text{GB}$，量化到 INT8 只要 $7\,\text{GB}$，INT4 只要 $3.5\,\text{GB}$，可以在消费级 GPU 上运行。量化带来的好处是三重的：减少显存占用、加速推理因为减少了访存量、以及省下的显存可以服务更多请求。质量损失通常很小，在 $1\%$ 到 $3\%$ 之间。

GPTQ：逐层最优量化

问题：直接四舍五入会累积误差。GPTQ 的思路：

$$\hat{W}^* = \arg\min_{\hat{W}} \left\| WX - \hat{W}X \right\|_2^2$$
  • 不是单独量化每个权重，而是考虑整层输出误差
  • 利用 Hessian 矩阵 $H = 2XX^T$ 指导量化
  • 逐列量化权重，同时调整未量化的列来补偿误差
  • 只需少量校准数据（~128 条）
与简单量化对比

Simple RTN: round-to-nearest，忽略层间关系
GPTQ: 考虑权重间的相关性，量化后层输出更接近原始

其他量化方法

AWQ (Activation-aware): 基于激活值重要性保护关键权重通道

GGUF: 支持 CPU 推理的量化格式，可在笔记本上运行 LLM

SmoothQuant: 将激活值的量化难度迁移到权重上

实践建议

INT4 量化通常损失 < 1% 精度
推荐：GPTQ/AWQ 用于 GPU，GGUF 用于 CPU

简单的四舍五入量化会累积误差，GPTQ 提出了更好的方案。

它的目标是最小化量化后整层输出的误差，公式在左边：寻找一个量化后的权重矩阵，使得它乘以 X 和原始权重乘以 X 尽可能接近。关键技巧是利用 Hessian 矩阵来指导量化——它告诉我们哪些权重对输出影响更大，需要更小心地量化。方法是逐列量化，每量化完一列，就调整还没量化的列来补偿误差。

实践中只需要 $128$ 条校准数据，量化后精度损失极小，不到 $1\%$。

右边列出了其他常见的量化方法：AWQ 基于激活值的重要性来保护关键通道；GGUF 支持 CPU 推理，可以在笔记本电脑上运行大模型；SmoothQuant 则是把激活值的量化难度迁移到权重上。总体建议是：GPU 推理用 GPTQ 或 AWQ，CPU 推理用 GGUF。

Flash Attention：分块计算

接下来看一个系统层面的优化——Flash Attention。

标准 Attention 的显存瓶颈在哪里？在计算 softmax 时，要先算出一个 $N \times N$ 的完整注意力矩阵。$N=8192$ 时，这个矩阵有 $6700$ 万元素，单层单头就要 $128\,\text{MB}$。这显然不可持续。

Flash Attention 的创新在于分块计算。看右边的示意图：不再是完整的大矩阵，而是把 $Q$、$K$、$V$ 分成小块，在 GPU 的快速 SRAM 缓存里计算局部注意力，然后累积结果。关键是不需要把完整的 $N \times N$ 矩阵写回慢速的 HBM 显存。

数学结果和标准 Attention 完全一致，但显存从 $O(N^2)$ 降到了 $O(N)$。实际效果是在长序列推理时速度提升 $2$ 到 $4$ 倍，显存大幅减少。这是一个非常重要的优化，几乎所有现代 LLM 推理框架都默认使用 Flash Attention。

PagedAttention & 分布式推理

PagedAttention (vLLM):

  • KV Cache 分成固定大小的页/块
  • 像 OS 虚拟内存一样管理
  • 消除显存碎片
  • 支持 KV Cache 共享（prefix caching）

Tensor Parallelism:

  • 每层权重切分到多张 GPU
  • $70\text{B}$ 模型：$4\times$ A100 ($80\,\mathrm{GB}$)
  • 通信开销：All-Reduce 操作
PagedAttention 是 vLLM 框架的核心创新，灵感来自操作系统的虚拟内存管理。

请看左边两张图的对比。上面是传统的 KV Cache 分配方式：给每个请求预分配一段连续的显存空间。问题是有些请求用不完预分配的空间，造成碎片浪费。下面是 PagedAttention 的做法：把 KV Cache 分成固定大小的块，像操作系统的内存页一样管理。不同请求的 KV Cache 可以不连续存储，通过一个页表来映射。这样就消除了碎片问题，显存利用率从大约 $60\%$ 提升到 $90\%$ 以上。

下方还提到了分布式推理。当模型大到单张 GPU 放不下时，比如 $70\text{B}$ 模型需要 $4$ 张 A100，有两种并行策略：Tensor Parallelism 是把每一层的权重切分到多张 GPU 上同时计算；Pipeline Parallelism 是把不同的层放到不同的 GPU 上流水线处理。通常两者结合使用。

推理加速技术全景

这张全景图总结了我们介绍的所有推理加速技术，分为四个方向。

第一，算法优化：核心就是 Speculative Decoding，用小模型帮大模型做投机执行。

第二，模型压缩：量化是最常用的手段，包括 GPTQ、AWQ 等方法。此外还有剪枝和蒸馏，今天没有展开。

第三，系统优化：Flash Attention 解决了 Attention 的显存瓶颈，PagedAttention 优化了 KV Cache 的管理，Continuous Batching 提升了服务吞吐量。

第四，硬件加速：通过 Tensor Parallelism 和 Pipeline Parallelism 实现多 GPU 分布式推理。

这些技术通常组合使用。比如 INT4 量化加上 Flash Attention 再加上 Speculative Decoding，可以获得累积的加速效果。

Inference-Time Scaling

用更多推理计算换更好结果
Part 2 结束，过渡到 Part 3。主讲人 C 接力。 过渡语：单次推理变快后，能否用更多推理计算提升效果？

两种 Scaling 范式

训练 Scaling（更大模型）vs 推理 Scaling（更多思考）— 两种提升性能的方式

大家好。前两部分我们关注的是如何让推理更快。现在换一个视角：如果我们愿意在推理时花费更多的计算，能否得到更好的结果？这就是 Inference-Time Scaling 的核心思想。

请看这张对比图。左边是训练时的 Scaling：用更大的模型、更多的数据、更多的训练算力，来让模型变得更强。这是我们比较熟悉的方式。右边是推理时的 Scaling：保持模型不变，但在推理时使用更多的计算。比如让模型思考更多步、采样更多次、做更深入的搜索。

打个比方：训练 Scaling 相当于让学生学更多年、读更多书；推理 Scaling 相当于考试时让学生想得更久。两者都能提升表现，但成本和方式不同。

今天这个第三部分，我们就来探讨推理 Scaling 的各种技术。

Chain-of-Thought (CoT)

❌ 直接回答

Q: 一个餐厅有 23 张桌子。每张桌子有 4 把椅子。今天午餐时有 14 张桌子坐了 3 人，5 张桌子坐了 4 人，其余空着。餐厅里有多少人？

A: 62 人

（一步跳到答案，容易出错）

✅ 分步推理 (CoT)

Q: （同上）

A: 让我一步步来：

1) $14 \times 3 = 42$ 人

2) $5 \times 4 = 20$ 人

3) 其余 $23-14-5=4$ 张桌子空着

4) 总人数 $=42+20=62$ 人

最简单的推理 Scaling 方法就是 Chain-of-Thought，也就是思维链。

请看屏幕上的对比。左边是直接回答：模型看到问题后，一步就给出答案 62 人。问题是，模型要在"脑子里"一步完成所有计算，很容易出错。

右边是使用 CoT 的方式：模型说"让我一步步来"，然后一步步列出计算过程——14 乘以 3 等于 42 人，5 乘以 4 等于 20 人，剩下 4 张桌子空着，总共 42 加 20 等于 62 人。

两种方式得到的答案一样，但 CoT 的每一步都很简单，不容易出错。而没有 CoT 时，模型需要一步映射，中间任何计算错误都无法纠正。

CoT 的关键洞察是：将一个复杂问题分解成多个简单步骤。每步只消耗少量推理计算，但总计算量增加了。这就是"用更多推理计算换更好结果"的最简单形式。

CoT 为什么有效？— 数学直觉

直接回答 vs 分步推理：

$$P(a \mid q) = \sum_{c} P(a \mid c, q) \cdot P(c \mid q)$$
  • $q$ = 问题，$a$ = 答案，$c$ = 中间推理步骤
  • 直接回答需要模型一步映射 $q \to a$
  • CoT 引入中间变量 $c$，将一步分解为多步
  • 每步更简单 $\to$ 出错概率更低
Circuit Complexity 视角

CoT 等价于增加神经网络的"计算深度"。更深的网络能表达更复杂的函数。每个推理步骤 $\approx$ 网络多一层。

CoT 为什么有效？我们可以从数学角度来理解。

请看左边这个公式。$P(a \mid q) = \sum_{c} P(a \mid c, q) \cdot P(c \mid q)$，直接回答时，模型要一步从问题 $q$ 映射到答案 $a$，这是一个很复杂的函数。引入 CoT 后，我们把这个映射分解为多步：$q \to c_1 \to c_2 \to \cdots \to a$。每一步都是一个更简单的函数，出错的概率更低。

右边用图来直观展示。上面一行是直接回答，只有一层"网络深度"。下面一行是 CoT，有多步推理，相当于网络有 $T$ 层。

这就引出了 Circuit Complexity 的视角：CoT 本质上增加了神经网络的"计算深度"。理论上有一些问题，浅层网络需要指数级的宽度才能解决，但深层网络可以高效处理。CoT 让模型在推理时获得了额外的深度，这就是 Inference-Time Compute 的本质。

Self-Consistency：多次采样 + 多数投票

CoT 是一条推理路径。Self-Consistency 是它的自然延伸：不只做一次推理，而是采样多次。

请看这张图。同一个问题，我们采样 $N=5$ 次。每次走不同的推理路径，得到不同的答案。路径 1 得到 62，路径 2 得到 62，路径 3 得到 58，路径 4 得到 62，路径 5 得到 65。然后做多数投票：62 有 3 票，58 有 1 票，65 有 1 票。最终选择 62 作为答案。

背后的直觉是：正确的推理路径可能各不相同，但更可能殊途同归，得到同一个正确答案。而错误的路径则是随机分散的，不太可能凑到同一个错误答案上。

计算开销是 $5$ 次推理，等于 $5\times$ 的计算量，但准确率显著提升。这就是 Inference-Time Scaling 的典型例子：花更多计算换取更好的结果。

Self-Consistency：投票效果分析

Inference-Time Scaling 的核心直觉

更多采样 $\times$ 投票 = 更可靠答案。
计算量翻倍 → 准确率持续提升，但存在收益递减。

上一页展示了 Self-Consistency 的原理，这页用量化数据来看投票效果。

以 LLaMA-2-70B 在 GSM8K 数学推理基准上的表现为例，单次推理准确率是 $57\%$。通过多数投票，$N=5$ 时准确率提升到约 $63\%$，$N=11$ 约 $68\%$，$N=21$ 约 $74\%$，$N=41$ 约 $82\%$。

但注意边际收益递减：从 $1$ 次到 $11$ 次提升了 $11$ 个百分点，从 $11$ 次到 $41$ 次只提升 $14$ 个百分点，而计算量增加了近 $3$ 倍。

还有一个重要前提：多数投票要求单次准确率 $p > 0.5$ 才有效。这来自 Condorcet 陪审团定理——如果每个"投票者"正确的概率超过 $1/2$，投票者越多，最终正确的概率就越高。反之如果单次准确率 $p < 0.5$，投票反而会更差。

总的来说，更多采样加投票等于更可靠的答案，但存在收益递减，需要权衡计算成本。

Best-of-N & Process Reward Model

Best-of-N：

  • 生成 N 个候选解法
  • 用 Reward Model 打分
  • 选分数最高的
$$E[\text{quality}] = E\!\left[\max_{i=1}^{N} f(x_i)\right]$$

Process Reward Model (PRM):

  • 不只评估最终答案
  • 评估每一步推理的质量
  • 可以定位出错的步骤
  • OpenAI o1 的核心技术之一
Best-of-N 是一种更通用的"多采选优"方法。

看左边的流程图。生成 $5$ 个候选解法，然后交给一个 Verifier 或者 Reward Model 来打分，最后选分数最高的那个。数学上，期望的最优质量会随 $N$ 的增长而增长，当然收益也是递减的。

但更精细的方法是 Process Reward Model，简称 PRM。普通的 Verifier 只看最终答案对不对，而 PRM 评估推理过程中每一步的质量。它可以定位到具体哪一步出了错，从而指导搜索算法在正确的方向上探索。

PRM 是 OpenAI o1 模型的核心技术之一。通过评估每一步的推理质量，模型可以在思考过程中自我纠错，避免在一个错误的方向上越走越远。

Tree of Thoughts & MCTS

从线性思考到树状搜索：

  • CoT: 一条推理路径
  • ToT: 多条分支 + 回溯
  • MCTS: AlphaGo 式搜索
$$\text{UCB} = \bar{X}_j + c\sqrt{\frac{\ln N}{n_j}}$$
  • $\bar{X}_j$ = 节点 $j$ 的平均奖励
  • $N$ = 父节点访问次数
  • $n_j$ = 节点 $j$ 的访问次数
  • $c$ = 探索-利用平衡系数

UCB 保证：探索未知路径 + 利用已知好路径

Tree of Thoughts 将推理从线性的 CoT 升级为树状搜索。

CoT 是一条路走到黑。Tree of Thoughts 允许在每一步分支出多个候选思路，评估后选择最有希望的继续探索，不行的就回溯。MCTS，也就是蒙特卡洛树搜索，来自 AlphaGo，核心用 UCB 公式来平衡探索和利用。

大家看这个 UCB 公式。$\bar{X}_j$ 是节点 $j$ 的平均奖励，越高说明这条路越好。但后面这一项中，大 $N$ 是父节点的访问次数，小 $n_j$ 是当前节点的访问次数。如果一个节点被访问得很少，这一项就会很大，给它更多被选择的机会。这就是"探索"——避免错过潜在的好路径。

实际效果是：在数学推理、编程等复杂任务上，Tree of Thoughts 显著优于简单的 CoT，因为复杂问题往往需要试错和回溯。

OpenAI o1/o3：Test-Time Compute 的工业实践

o1 的核心思路
  • 训练模型"学会思考"（强化学习训练推理策略）
  • 推理时自动进行多步推理 + 自我验证
  • 内置 hidden CoT（思考过程对用户不可见）
  • 在数学/编程/科学推理上大幅超越 GPT-4

  • 推理计算量可动态调整（简单问题少想，难题多想）

o1/o3 证明：推理时的计算扩展是提升 LLM 能力的新维度，与模型规模扩展互补。

OpenAI 的 o1 和 o3 模型是 Inference-Time Scaling 最著名的工业实践。

左栏总结了 o1 的核心思路。它用强化学习训练模型"学会思考"——也就是自动进行多步推理、自我验证和纠错。推理时，模型会先生成一段 Hidden Chain of Thought，对用户不可见，然后给出最终答案。

这验证了 Inference-Time Scaling 的有效性：不一定要更大的模型，让模型思考更久也能达到更好的效果。

$$\text{Performance} \propto \log(\text{Inference FLOPs})$$
  • 推理计算量与性能呈对数线性关系
  • 与训练 scaling law 形式类似（$\text{Loss} \propto \log(\text{Compute})$）
  • 小模型 + 更多推理计算 可以匹配大模型的性能
  • 存在"计算最优"的分配策略
实践意义

不一定需要训练更大的模型。对于单个困难问题，增加推理计算（更多采样、搜索）可以显著提升准确率。

最后来看 Inference Scaling Laws。

核心发现是：推理计算量与模型性能呈对数线性关系，形式上和训练时的 Scaling Law 非常类似。看右边的图：横轴是推理计算量的对数，纵轴是准确率，基本上是一条直线。

更有趣的发现是：一个 $7\text{B}$ 的小模型，如果分配更多的推理计算，比如 $64$ 次采样加验证，可以达到 $70\text{B}$ 大模型单次推理的性能。这意味着：对于单个困难问题，用小模型多算几次可能比用大模型算一次更划算——更快也更省资源。

当然，这也提示了一个计算最优的分配问题：给定固定的总预算，应该花多少在训练上、多少在推理上？这是一个活跃的研究方向。
