从序列模型到 Attention

大家好，欢迎来到今天的课堂。今天我们的主题是大语言模型的推理，内容分为三个部分：首先我会带大家回顾 Transformer 的推理流程，然后第二部分介绍推理加速技术，第三部分探讨一个有趣的问题——能不能通过增加推理时的计算量来提升模型效果。

先来看这张对比图。左边是我们熟悉的 RNN，处理序列信息时必须一个一个地传递，信息从 $h_1 \to h_2 \to h_3$，依次串行传递。这就像传话游戏，第一个人的信息要一个个传到最后，中间很容易丢失，训练时还会遇到梯度消失的问题，长距离依赖关系很难学到。

右边是 Transformer 的核心创新——Attention 机制。它的关键在于：每个位置可以直接"看到"所有其他位置，不需要串行传递。就像所有人同时在一个会议室里，谁想说什么直接说，不需要传话。这种并行性是后面推理加速的重要基础。

Self-Attention 机制

核心公式：

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$
  • Q (Query): 当前 token 想了解什么
  • K (Key): 每个 token 能提供什么信息
  • V (Value): 每个 token 的实际内容
  • $\sqrt{d_k}$ 缩放防止 softmax 饱和
接下来我们详细拆解 Attention 的计算过程。请看屏幕上的公式 $\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$。

为了方便理解，我们用一个图书馆的类比。左边这张流程图下面也画了这个类比。想象你去图书馆找资料，你在搜索框输入的关键词就是 Query——你想了解什么。每本书上贴的标签和索引就是 Key——这本书能提供什么信息。书的实际内容就是 Value。

计算过程是这样的：首先 $Q$ 和 $K$ 做内积，得到一个"相关性分数"——你的关键词和每本书的标签有多匹配。然后过一遍 softmax，把这些分数变成权重，加起来等于 1。最后用这些权重对 $V$ 做加权求和，得到最终输出。

公式里除以 $\sqrt{d_k}$ 这一步很重要：当维度 $d_k$ 比较大时，$QK^T$ 的内积数值会很大，导致 softmax 输出接近 one-hot，梯度几乎为零。除以 $\sqrt{d_k}$ 可以缓解这个问题。

Multi-Head Attention

为什么不只用一个 Attention？

  • 每个 Head 学习不同的注意力模式
  • Head 1 可能关注语法关系
  • Head 2 可能关注语义相似性
  • Head 3 可能关注位置关系...
关键参数

LLaMA-2-7B: 32 heads, $d_{model}=4096$, $d_{head}=128$

理解了单个 Attention 之后，一个自然的问题是：为什么不只用一个？为什么要搞多头？

答案在左边这张图上。每个 Head 有自己独立的 $Q$、$K$、$V$ 权重矩阵，独立计算 Attention，最后把所有头的结果拼接起来，通过一个线性层投影回去。

关键在于：不同的 Head 会学到不同的注意力模式。比如 Head 1 可能关注语法关系，Head 2 关注语义相似性，Head 3 关注位置关系。通过多个"视角"来看同一句话，模型能捕获更丰富的信息。

举一个具体的例子：LLaMA-2-7B 有 32 个头，每个头的维度是 128，$32 \times 128 = 4096$ 正好等于模型的隐藏维度 4096。这个设计在后续理解推理的计算量时会用到。

Transformer Decoder Block

现在我们把 Attention 放到完整的 Transformer Decoder 里。请看这张结构图，从下往上看。

最下面是输入经过 Embedding 和位置编码。然后进入 Masked Multi-Head Self-Attention，注意这里是 Masked 的——每个位置只能看到自己和之前的位置，不能"偷看"后面的 token。这是因果性约束，也是后面自回归生成的基础。

接下来是 Add and LayerNorm——残差连接加层归一化。残差连接把输入直接加到输出上，缓解了深层网络的梯度消失问题。然后是 Feed-Forward Network，也就是 FFN，由两个线性层和一个激活函数组成，存储了模型大部分的参数。

这个 Attention 加 FFN 的结构会在上方虚线框里重复堆叠。比如 LLaMA-2-7B 就堆叠了 32 层，每一层都有自己独立的参数。最终通过一个线性层和 Softmax，输出词表上的概率分布。

自回归生成 (Autoregressive Generation)

核心思想：逐 token 生成，每次根据已有内容预测下一个

$$P(y_t \mid y_{\lt t}, x) = \text{softmax}(W \cdot h_t)$$
  • $x$ = 输入 prompt
  • $y_{\lt t}$ = 已生成的 tokens
  • $h_t$ = Decoder 最后一层的隐状态
  • 每次只生成一个 token，追加后继续
  • 遇到 [EOS] 或达到长度上限时停止
为什么慢？

生成 1000 个 token 需要跑 1000 次前向传播！

有了 Transformer Decoder 的结构，我们来看它是怎么生成文字的。

核心思想很简单：逐个 token 生成，每次根据已有的内容预测下一个。请看左边的示意图：第一步输入起始符 BOS，模型预测出"今"字。第二步把"今"追加进去，模型预测"天"。第三步继续，直到遇到结束符或者达到长度上限。

右边是这个过程的数学表达。$h_t$ 是 Decoder 最后一层输出的隐状态，通过一个线性层加 softmax，得到词表上的概率分布。选择概率最大的或者采样的 token 作为下一个输出。

请大家注意右下角这个提示：生成 1000 个 token 需要跑 1000 次前向传播。这就是 LLM 推理慢的根本原因——每生成一个词，整个模型都要完整跑一遍。记住这一点，后面讲加速技术时会用到。

Prefill vs Decode：推理的两个阶段

进一步细化推理过程，我们可以把它分成两个阶段：Prefill 和 Decode。

请看左边，Prefill 阶段。用户的整个 prompt 一次性送入模型，做的是矩阵乘法——一个 $N \times d$ 的输入矩阵乘以 $d \times d$ 的权重矩阵。这是典型的计算密集型操作，GPU 的算力可以充分利用。

再看右边，Decode 阶段。每次只有一个新 token 进来，做的是向量乘法——一个 $1 \times d$ 的向量乘以 $d \times d$ 的权重矩阵。计算量很小，但每次都要从显存里读取整个权重矩阵。所以瓶颈不在算力，而在显存带宽——这就是 Memory-bound。

打个比方：Prefill 像是快递员一次性送一批快递，效率很高。Decode 像是每次只送一个快递但都要跑一趟仓库取货，大部分时间花在路上。

KV Cache：避免重复计算

接下来看推理加速最基本也最重要的技术——KV Cache。

请看左半边，没有 KV Cache 的情况。$t=1$ 时计算 token a 的 $K$ 和 $V$。$t=2$ 时要把 a 和新的 token b 一起重新算 $K$ 和 $V$。$t=3$ 又要把 a、b、c 全部重算。虚线框里标红的都是重复计算——之前算过的 $K$ 和 $V$，每次又要重新算一遍，非常浪费。

右边是有了 KV Cache 的做法。$t=1$ 算出 $K_1$ 和 $V_1$，存入 cache。$t=2$ 只需要算新 token 的 $K_2$ 和 $V_2$，然后从 cache 里读取之前的 $K_1$ 和 $V_1$。$t=3$ 同理，只算 $K_3$ 和 $V_3$，从 cache 读取之前所有的 $K$ 和 $V$。

Trade-off 很明显：KV Cache 节省了大量重复计算，加速了推理，但代价是占用额外的显存来存储这些 $K$ 和 $V$ 向量。具体占多少呢？下一张 slide 来量化分析。

KV Cache 占多少显存？

$$M_{\text{KV}} = 2 \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times S \times \text{bytes}$$
参数含义LLaMA-2-7B
$n_{\text{layers}}$层数32
$n_{\text{heads}} \times d_{\text{head}}$注意力维度4096
$S$序列长度变量
bytes精度 (FP16)2
计算示例

LLaMA-2-7B, $S=4096$:
$M = 2 \times 32 \times 4096 \times 4096 \times 2$ $= 2\,\mathrm{GB}$

（图表由 Python 生成。模型参数来自 Touvron et al. 2023 / HuggingFace LlamaConfig）

KV Cache 到底占多少显存？请看这个公式。$M_{\text{KV}}$ 等于 $2 \times n_{\text{layers}} \times d_{\text{head}} \times n_{\text{heads}} \times S \times \text{bytes}$。其中 $2$ 是因为要同时存 $K$ 和 $V$ 两份。

我们用 LLaMA-2-7B 的具体参数来算。$n_{\text{layers}}=32$，注意力维度 $4096$，FP16 精度下每个元素 $2$ 字节。当序列长度 $S=4096$ 时，代入公式：$2 \times 32 \times 4096 \times 4096 \times 2 = 2\,\text{GB}$。

也就是说，仅仅是为了存储 KV Cache，序列长度 $S=4096$ 时就要 $2\,\text{GB}$ 显存。如果换成 70B 的模型，或者更长的序列，这个数字会更加夸张。

右边的柱状图直观展示了不同序列长度下的显存占用。大家可以看到，长文本场景下 KV Cache 会成为显存瓶颈。这就是为什么后面会有 PagedAttention 等技术来优化 KV Cache 的管理。

推理延迟分解

$$T_{\text{total}} = T_{\text{prefill}} + N \times T_{\text{decode}}$$
  • TTFT (Time To First Token) $= T_{\text{prefill}}$
    用户发 prompt 到看到第一个 token 的时间
  • TPOT (Time Per Output Token) $= T_{\text{decode}}$
    每生成一个 token 的平均时间
瓶颈在哪里？

对于长生成（$N$ 大），Decode 阶段 dominates，因为 Decode 是 memory-bound（受限于显存带宽）

（图表由 Python 生成，延迟值基于 LLaMA-2-7B FP16 在 A100 上的典型推理延迟）

有了 Prefill/Decode 和 KV Cache 的概念，我们可以精确地分析推理延迟了。

总延迟 $T_{\text{total}} = T_{\text{prefill}} + N \times T_{\text{decode}}$，这里 $N$ 是输出 token 的数量。两个关键指标：TTFT 也就是 Time To First Token，$= T_{\text{prefill}}$，代表用户发出 prompt 后等多久才能看到第一个字。TPOT 也就是 Time Per Output Token，$= T_{\text{decode}}$，代表每生成一个 token 要多久。

关键洞察在右边的瀑布图里。当生成 500 或 1000 个 token 时，Decode 阶段占据了绝大部分时间。因为 Decode 是 Memory-bound 的——每步都要从显存读取整个权重矩阵，但只做一次很小的向量乘法，GPU 算力大量闲置。

所以加速推理的重点，在于优化 Decode 阶段的 Memory-bound 问题。具体怎么优化呢？这就是 Part 2 要讲的内容。

LLM 推理全流程：从输入到输出

这张 slide 把前面讲的所有概念串成了一个完整的推理流程图，大家跟着我从左到右看一遍。

用户输入一段自然语言的 prompt，首先经过 Tokenization 转换成 token ID 序列。然后进入 Prefill 阶段，并行处理所有 prompt token，这一步是 Compute-bound 的矩阵乘法。接着进入 Decode Loop，逐个 token 生成，每步都是 Memory-bound 的向量乘法，KV Cache 逐步增长。生成结束后，经过 Detokenization 把 token ID 转回文本输出。

下方标注了三个关键性能指标：TTFT、TPOT 和总延迟。对于长生成场景，Decode 阶段 dominates，加速推理的重点就在这里。底部也预告了下一部分的优化技术：Speculative Decoding、量化、Flash Attention、PagedAttention。

Part 1 小结

核心概念
  • Attention: $QK^T/\sqrt{d_k} \cdot V$
  • 自回归: 逐 token 生成
  • KV Cache: 避免重复计算
关键洞察
  • Prefill $=$ Compute-bound
  • Decode $=$ Memory-bound
  • 长生成时 Decode dominates
